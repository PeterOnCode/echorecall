# Specification Plan

**Feature:** Batch text-to-speech generation, rich audio tagging, a managed library,
and an app **Settings** surface (theme, language, OpenAI key) — delivered on a
**three-tab** Nuxt UI interface (Generate / Library / Settings).

> Most decisions below are **ported from the echoquize project** (`specs/002-studio-enhancements`),
> which already specified this exact feature set for the Python/Gradio original. echorecall
> re-implements that behavior in TypeScript/Nuxt. Code is **not** copied (different stack); the
> behavior, rules, and design decisions are.
>
> The **Settings tab, theming/dark mode, i18n, and in-app OpenAI-key setup are new** to
> echorecall — there is no echoquize equivalent to port.

## Context

EchoRecall today converts a single block of text into one spoken MP3 (voice
selection only) and automatically saves it to a browsable, replayable library. This
feature evolves that single-shot tool into a productive batch workflow with rich,
editable audio metadata and a manageable library, and rebuilds the interface around
a tabbed layout. The audience is unchanged: a self-hosted, single user who wants to
produce many narrated audio files at once from a list, attach proper metadata so the
files are well-organized in any audio player, and rename/retag/delete what they have
generated — all without leaving the browser.

## Interface

Reorganize the app surface into three tabs: **Generate**, **Library**, and
**Settings**. Every raw HTML form control is replaced by its component-library
equivalent for a consistent, accessible UI. The current application version is
displayed next to the app title in the header. An app-wide **theme** (light / dark /
system) and **language** selector are reachable from the header and/or the Settings
tab; both preferences persist client-side (cookie) so SSR and reload stay consistent.

> i18n, theming, and the Settings surface are **web-surface only** — they live in the
> Nuxt adapter and MUST NOT leak into the framework-agnostic shared core (Constitution
> Principle IV). The OpenAI-key *resolution* (UI store → env fallback) is the exception:
> it lives in the shared core/server so a future CLI honors the same precedence.

## Generate tab — add a single item

A text area for one item with a live character counter shown against the maximum
allowed length (**4,096 characters**, carried over from the current app). The user
chooses the **voice, model, output format, and speed** at the form level, edits the
audio metadata tags, and **adds the text as one item to the generation list**. The
single text box and the `.txt` upload both feed the **same list**, and there is
**exactly one Generate button** for the whole list — there is no separate
one-shot/immediate generate. Inline preview/playback and download remain available per
generated result.

> Divergence from echoquize: echoquize kept an immediate single-generate path separate
> from its batch queue. echorecall deliberately **unifies** them — textbox and upload
> both populate one list, generated by a single button.

- Models: **`tts-1`**, **`tts-1-hd`**, **`gpt-4o-mini-tts`**. The **instructions**
  field is sent only when the model is `gpt-4o-mini-tts`.
- **Speed** is a form-level control applied to all items; it is **not** editable per
  queue row (see Resolved decisions Q1). *Implementation note:* echoquize forwards
  `speed` (0.25–4.0) for all models — verify whether `gpt-4o-mini-tts` actually honors
  it.

## Generate tab — batch from a file

The user can bulk-load the queue by uploading one `.txt` file (UTF-8, **under 5 MB**).
Each line becomes one queued audio item, in the file's original order; uploaded items
are **appended** to any existing queue (never replacing it). Each line is trimmed;
blank/whitespace-only lines are skipped; a line longer than 4,096 characters is
rejected with a message identifying it; valid lines are still added. After import the
user sees an accurate **added / skipped-blank / rejected-too-long** summary, with
exactly one queued item per valid line in order. The uploaded file is not persisted;
nothing is generated until the user starts generation. New items inherit the
voice/model/format/speed currently selected in the form and remain editable.

## Generate tab — editable queue

Every queued item is individually editable: its **text, voice, model, output format,
instructions, and audio tags**. Editing one row changes only that row and no other.
Empty or over-length (>4,096) text is rejected on edit, keeping the previous valid
value. The **instructions** value is retained when the item's model changes (but only
sent for `gpt-4o-mini-tts`). Changing a row to a non-taggable format (AAC/PCM) warns
that its tags will be skipped **without discarding** the entered values. Speed is not
editable per row.

## Generate tab — audio metadata (expanded ID3v2.4.0 tag set)

The Generate form lets the user edit the full tag set, written into each generated
file and applied per-item at batch generation. Fields:

- **Title** (also the filename source), **Artist**, **Album**, **Genre**, **Comment**
- **Recording date/time** — accepts a full timestamp *or* year-only
- **Track number** — `n` or `n/total`
- **Language(s)** — ISO 639-2 codes, multi-value
- **Custom text** entries — repeatable, description-keyed
- **Custom URL** entries — repeatable, description-keyed

Per-format behavior:

- **MP3 / WAV** → written as **ID3v2.4.0** (full set).
- **FLAC / Opus** → native Vorbis/Opus comments for fields with an equivalent; fields
  without one (e.g. custom URL) are skipped with a notice.
- **AAC / PCM** → untaggable; tagging is skipped with a notice and the generation
  still completes.

Tag values are persisted in the database (authoritative store) so they display and
re-edit later; saving replaces the full tag set (clearing a field removes it).

## Generate tab — generate and output

A single Generate button synthesizes audio for every queued item, applying each
item's own settings and tags. Each output file is **named from its title**:

- Slug rule: transliterate to ASCII, lowercase, whitespace→`_`, remove characters
  outside `[a-z0-9_-]`, collapse separators, cap at **64 chars**, then append the
  format extension.
- An empty title (or one that slugifies to nothing) falls back to a unique identifier
  so a valid filename always exists.
- **Collision-safe**: a numeric suffix (`…_2`) is appended on collision, scoped to the
  day folder; an existing file is never overwritten.

Generated audio is stored under a **`YYYY/MM/DD`** folder derived from the UTC
creation date. The stored record references the actual saved path. Every generation is
saved to the library. When a multi-item batch is generated, the clips are also offered
as a single downloadable **`.zip`**, in addition to per-item save/download.

## Generate tab — default tag values

Tag fields on the Generate form (and the tags of newly added queue items) are
pre-filled from **deployment-configured default values** (e.g. a fixed artist/album),
read from the environment — not an in-app settings screen. Defaults never force a
value: the user can override or clear any field per generation, and the **Title is
never defaulted**. Missing/invalid configuration falls back to blank fields rather than
failing at startup.

## Library tab

Lists all previously generated items. From the library the user can:

- **Rename** a saved file (manual rename validated by the same slug rules + 64-char
  cap + collision suffix; reports the final name; empty/un-sluggable names are
  rejected, keeping the original; the extension is not editable; editing the *title*
  tag does **not** auto-rename the file).
- **Edit its full ID3v2.4.0 tag set** (the Library editor is the single place to edit a
  saved item's tags; untaggable formats keep the filename editable while tag writing is
  skipped).
- **Delete** an item (removing both the entry and its stored audio).
- **Search** the library by free text over **title, source text, and tags** (and
  filename), combinable with the filters below.
- **Sort** by **created date** (default, newest-first), **title**, **voice**, or
  **format**, ascending or descending.
- **Browse** with pagination and filtering (e.g. by date/voice), and **bulk-clean**
  multiple items at once by date/voice.

Search, sort, filter, and pagination are evaluated **server-side over SQLite** (so they
compose and scale as the library grows) and are **core capabilities** exposed through
the shared core — reachable from the CLI too, not just the web UI (Constitution
Principle IV).

This intentionally makes saved entries' **filename and tags mutable** (relaxing the
prior "entries are immutable" rule); the generated audio content / source text is still
not re-synthesized. **Per-item re-generate is intentionally out of scope** for this
feature — producing different audio means starting a new generation.

## Settings tab

A third tab for app-wide configuration, persisted appropriately per setting.

### Appearance (theming / dark mode)

A **light / dark / system** color-mode selector (Nuxt UI color-mode). The default
follows the OS (**system**); the choice persists client-side (cookie) so it survives
reloads and is correct during SSR (no theme flash). Applies across all tabs.

### Language (i18n)

UI labels/chrome are localized to **English** and **Hungarian**, with **Hungarian as
the default** locale; the user can switch language from Settings (and/or the header),
persisted client-side. Only interface text is translated — **generated audio, the
user's input text, tag values, and filenames are never translated**. All user-facing
strings are externalized into locale message catalogs (no hard-coded UI copy).

### OpenAI API key

- The Settings tab lets the user **set / replace / clear** the OpenAI API key in-app.
  The key is **write-only from the UI**: submitted to the server, stored **encrypted
  at rest** server-side, and **never read back to the client** — the UI shows only a
  masked status (e.g. "configured · ●●●●1234") plus a **Test key** action.
- **Precedence: a UI-set key takes precedence over the `OPENAI_API_KEY` environment
  variable** (your choice); `.env` is the **fallback** used only when no UI key is set,
  so existing env-based deployments keep working. Clearing the UI key reverts to env.
- **Server-only**: never sent to the client bundle, never written to logs, never echoed
  in errors. Because the key can change at runtime, the TTS provider resolves the active
  key **per request** (UI store → env fallback) rather than binding once at startup;
  that resolution lives in the shared core/server so the CLI honors the same rule.
- **Storage**: a new server-side settings store (e.g. an `app_config` row or an
  encrypted file under the mounted `data/` volume), encrypted with a server secret
  supplied via env — separate from the `generations` data.

> ✅ **Constitution amended (v2.4.0, 2026-06-17).** The Technology Stack secret-handling
> clause now permits a UI-set OpenAI key persisted **encrypted at rest, server-only,
> never logged**, taking precedence over the env var which remains a fallback. This
> capability is therefore constitution-compliant; no further governance step is needed.

## Explicit technical constraints (user-required)

Use the Nuxt UI component library for all interface elements (including its Tabs
component for the two-tab layout). Add the **Bumpp** tool to bump the version, commit,
and tag in one command, against a single authoritative version (the package version);
the UI reads that same value at runtime and shows it next to the title, degrading
gracefully (omit/placeholder) if it can't be determined. (echoquize used Python's
`bump-my-version` + `importlib.metadata`; echorecall's TS equivalent is Bumpp +
`package.json`.)

## Resolved decisions (ported from echoquize `002-studio-enhancements`)

1. **Speed** — exposed as a form-level control applied to all items; **not** editable
   per queue row. (This adds speed as a user control vs. today's fixed default.)
2. **Immutability** — deliberately relaxed: a saved entry's **filename and tags are
   editable** from the Library; the audio/text is never re-synthesized.
3. **Expanded tag set + slug rule** — defined above (Audio metadata + generate/output).
4. **Models / formats** — models `tts-1`, `tts-1-hd`, `gpt-4o-mini-tts` (instructions
   only for `gpt-4o-mini-tts`); formats MP3/WAV (ID3v2.4.0), FLAC/Opus (native, custom
   URL skipped), AAC/PCM (untaggable, skipped gracefully).
5. **"Edit audio ids"** — means editing the full ID3 tag set + filename from the
   Library, not the entry's unique identifier.
6. **Migration** — title-slug naming and `YYYY/MM/DD` storage apply to **new files
   only**; existing entries (UUID filenames, flat `data/audio/<id>.mp3` path) stay
   readable in place. The DB schema grows: add a stored **path** column (the path is no
   longer derivable from the id) **and** new **tag columns** (per-field `tag_*` plus a
   `tags_extra` blob for repeatable custom text/URL entries). Existing 001 rows are
   backfilled — path from `audio/<id>.mp3`, tags empty — so they keep working. Library
   rename acts on a single file on explicit request.

## Confirmed scope (echoquize parity + your additions)

1. **Output-format catalog** — full set: **MP3, WAV, FLAC, Opus, AAC, PCM** (exercises
   every tag path: ID3v2.4.0, Vorbis, and graceful skip).
2. **Default tag values** from deployment env — **in scope** (see "default tag values"
   above).
3. **Batch ZIP download** — **in scope** (single `.zip` per multi-item batch).
4. **Library extras** — **in scope**: pagination, filtering, and bulk-clean by
   date/voice.
5. **File-size cap** — the uploaded `.txt` is capped at **5 MB** (your choice;
   echoquize itself enforces no cap).
6. **Voice catalog** — widened to echoquize's full set: `alloy, ash, ballad, coral,
   echo, fable, onyx, nova, sage, shimmer, verse, marin, cedar` (vs. the smaller 001
   set).
7. **Theming** — light / dark / **system** color-mode, system default, cookie-persisted,
   applied app-wide (web surface only).
8. **i18n** — **English + Hungarian** UI locales, **Hungarian default**, switchable and
   persisted; interface text only (never the generated audio/content).
9. **Settings tab** — third tab hosting Appearance, Language, and OpenAI-key setup.
10. **In-app OpenAI key** — UI-set key persisted **encrypted at rest**, **UI precedence
    over `.env`** (env = fallback), write-only/masked, server-only, never logged.
    **✅ Permitted by constitution v2.4.0.**
11. **Library search** — free-text over title, source text, and tags (and filename),
    server-side, composes with filters.
12. **Library sorting** — by created date (default), title, voice, or format, asc/desc,
    server-side.

**Out of scope (this round):** per-item re-generate (queue or Library) — saved audio is
never re-synthesized; different audio means a new generation.
